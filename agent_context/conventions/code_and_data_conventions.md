# Code and Data Conventions

## Current State
These conventions are partly implied by the existing files, but most are not yet enforced by code.

### What the code currently does
- Loads ARC task JSON from a path supplied in the script.
- Iterates over `train` and `test` examples.
- Expects each example to contain `input` and `output` grids.
- Converts each grid to a flat list and records its dimensions.
- Writes a Parquet file with `pandas.to_parquet(..., engine="pyarrow")`.

### Current limitations
- No validation layer checks that the JSON matches ARC-AGI expectations.
- No schema contract guarantees column names, types, or nullability.
- No run metadata is captured beyond the task id in the transformed rows.
- No error handling exists for missing files, malformed JSON, or inconsistent grid shapes.

## Code Conventions
- Keep pipeline code inside `src/atlas_arc_failure/`.
- Prefer small modules over a single large script once the project starts to grow.
- Keep CLI code thin and delegate work to importable functions.
- Use explicit names for model, task, run, and artifact identifiers.
- Prefer deterministic logic and avoid hidden global state.

## Data Conventions
- Treat ARC task JSON as raw input, not as an analysis-ready format.
- Preserve raw data under `data/raw/` if it is copied into the repo workspace.
- Use `data/interim/` for intermediate, disposable transformations.
- Use `data/processed/` for validated, analysis-ready tables.
- Use `artifacts/` for outputs generated by experiments, runs, and analysis.

## Parquet Conventions
- Parquet should be the canonical output format for normalized tables and inference results.
- Each table should have a documented schema before it becomes a dependency for downstream work.
- Record explicit column types for task ids, example indices, phase, grid shape, payload data, timestamps, and run identifiers when those fields are added.
- Partitioning strategy should be intentional and documented before it is implemented.

## Prompt Conventions
- Store prompt templates separately from prompt versions and eval sets.
- Version prompts rather than overwriting them in place.
- Include the model or task context required to reproduce a prompt run.
- Keep prompts readable by both humans and CLI agents.

## Observability Conventions
- Capture run id, task id, model name, prompt version, start/end timestamps, latency, retries, and cost where available.
- Log failures with enough detail to reproduce the run locally.
- Separate application logs from experiment artifacts.

## Reproducibility Conventions
- Prefer pinned dependencies once the dependency list stabilizes.
- Do not rely on untracked local state for correctness.
- Make configuration explicit through files in `configs/` or environment variables documented in `.env.example`.
- Ensure every analysis result can be traced back to a specific input, prompt version, and model configuration.

## Assumptions
- The current `src/atlas_arc_failure/ingestion/main.py` script is a temporary proof of concept.
- The project will eventually replace script-style execution with importable modules and a CLI.
